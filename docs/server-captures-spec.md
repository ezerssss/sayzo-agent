# Eloquy Server Captures Spec — Context for Implementation

> **What this is:** A spec for the capture upload and processing pipeline that the Eloquy desktop agent expects. Drop this into the Claude session working on your Next.js app so it implements the correct server-side routes and processing.

## Background

Eloquy has a **local Python agent** that runs 24/7 on the user's machine. It captures real English conversations the user participates in (meetings, calls, etc.), transcribes them locally, filters out noise (YouTube, music, monologues), and uploads the kept captures to the server. The server's job is to:

1. Receive and store the raw capture (audio + transcript)
2. Re-transcribe with a better model (the agent uses Whisper small on CPU — good enough for filtering, not for coaching)
3. Validate relevance with a stronger model (the agent's local 3B LLM catches most junk, but edge cases slip through)
4. Deeply analyze the conversation for coaching value
5. Build a living profile of the user's English speaking patterns
6. Generate drill suggestions targeting the user's specific weaknesses
7. Enable scenario replay drills from real recordings

**The agent is already implemented.** The upload endpoint below is what the agent calls. Do not change the request contract — implement it exactly as specified.

**Auth is already implemented.** See `docs/server-auth-spec.md` for the auth endpoints. The upload endpoint uses the same JWT Bearer token described there.

## Stack

- **Next.js** (App Router)
- **Firebase Auth** — Google provider (already set up for auth endpoints)
- **Firebase Admin SDK** — verify JWTs, access Firestore
- **Firestore** — capture metadata, user profiles, drill suggestions
- **Cloud Storage (Firebase Storage)** — audio files
- **Cron + internal API routes** — async processing pipeline (see Processing Architecture below)
- **Paid STT API** — OpenAI Whisper API or Deepgram Nova-2 for server-side re-transcription
- **LLM API** — GPT-4o-mini, Claude Haiku, or similar for relevance validation + deep analysis

## What the Agent Uploads

### Audio File

- **Format:** Stereo Opus in OGG container
- **Bitrate:** 64 kbps (nominal)
- **Sample rate:** 16 kHz
- **Channels:** Stereo — **left = microphone (user's voice), right = system audio (other participants)**
- **Typical size:** 90 KB - 5 MB (5-60 minute conversations)
- **Already compressed** — no re-encoding needed on the server. Store as-is.

### Record JSON

The agent sends a JSON-serialized `ConversationRecord` with this exact shape:

```json
{
  "id": "5dcdc3cdc053",
  "started_at": "2026-04-10T12:32:02.536488+00:00",
  "ended_at": "2026-04-10T12:38:45.411488+00:00",
  "title": "Sprint planning with backend team",
  "summary": "The user discussed the new caching strategy with two team members. They assigned tasks for the Redis migration and agreed on a Thursday deadline.",
  "transcript": [
    {
      "speaker": "user",
      "start": 14.48,
      "end": 21.0,
      "text": "So I was thinking we should probably migrate to Redis before we do anything else"
    },
    {
      "speaker": "other_1",
      "start": 21.5,
      "end": 28.3,
      "text": "Yeah that makes sense. The current cache is basically falling over at peak."
    },
    {
      "speaker": "user",
      "start": 28.8,
      "end": 35.2,
      "text": "Exactly. And like, the latency is just, you know, unacceptable at this point"
    },
    {
      "speaker": "other_2",
      "start": 36.0,
      "end": 42.1,
      "text": "I can take the connection pool setup if you handle the serialization layer"
    }
  ],
  "audio_path": "audio.opus",
  "relevant_span": [0.0, 393.2],
  "metadata": {
    "close_reason": "joint_silence"
  }
}
```

### Field Reference

| Field | Type | Description |
|---|---|---|
| `id` | string | 12-char hex ID generated by the agent. Unique per capture. |
| `started_at` | ISO 8601 | When the conversation session opened (UTC). |
| `ended_at` | ISO 8601 | When the session closed (UTC). |
| `title` | string | Short title generated by the agent's local LLM. |
| `summary` | string | 1-3 sentence summary from the agent's local LLM. |
| `transcript` | array | Speaker-tagged, timed transcript lines (see below). |
| `audio_path` | string | Always `"audio.opus"` — ignore this field server-side. |
| `relevant_span` | [float, float] | Start/end in seconds. **Treat as a hint only** — this is the agent's local 3B LLM's best guess at the interesting part, padded by 15 seconds. Full uncropped audio is always included. The server should make its own relevance judgment with a better model. |
| `metadata.close_reason` | string | Why the session ended: `"joint_silence"` (45s of silence from both sides), `"safety_cap"` (hit 1-hour max), or `"shutdown"` (agent stopped gracefully). |

### Speaker Labels

| Label | Meaning |
|---|---|
| `"user"` | Audio from the microphone that matched the user's enrolled voiceprint. This is the Eloquy user — the person being coached. |
| `"other_unmic"` | Audio from the microphone that did NOT match the user's voiceprint. This means someone else was physically near the mic (e.g., in-person meeting). |
| `"other_1"`, `"other_2"`, ... | Audio from the system stream (other side of a call). Speakers are clustered by voice similarity, up to 4 distinct speakers. |

**Important:** The `"user"` label is the coaching target. All analysis should focus on how the user speaks. Other speakers provide conversational context but are not being coached.

## Endpoint to Implement

### `POST /api/captures/upload`

**Purpose:** Receive a capture from the agent after a conversation is detected and processed locally.

**Auth:** `Authorization: Bearer {access_token}` — verify the JWT as described in `server-auth-spec.md`. Extract the Firebase UID from the `sub` claim.

**Content-Type:** `multipart/form-data`

**Request fields:**

| Field | Type | Description |
|---|---|---|
| `record` | string (form field) | JSON-serialized ConversationRecord (shape shown above) |
| `audio` | file | `audio.opus` — stereo Opus/OGG binary |

**Success response (201 Created):**
```json
{
  "capture_id": "firestore-doc-id",
  "status": "queued"
}
```

**Error responses:**
```json
// 400 Bad Request
{"error": "invalid_record", "message": "Missing required field: transcript"}

// 401 Unauthorized (auth middleware)
{"error": "unauthorized", "message": "Invalid or expired token"}

// 413 Payload Too Large
{"error": "payload_too_large", "message": "Audio file exceeds 50MB limit"}

// 500 Internal Server Error
{"error": "internal", "message": "Failed to store capture"}
```

**Endpoint behavior:**
1. Verify JWT, extract Firebase UID.
2. Parse the `record` JSON field. Validate required fields exist.
3. Upload the audio file to Cloud Storage at `captures/{uid}/{capture_id}/audio.opus`.
4. Create a Firestore document at `captures/{capture_id}` with the record metadata, `uid` field (from JWT), and `status: "queued"`.
5. Return 201 immediately — do NOT wait for processing. The cron-based pipeline picks it up.

## Processing Architecture

Processing is **async** — a cron job picks up queued captures and processes them through an internal API route. This keeps everything inside the Next.js app with no extra infrastructure. Since the server runs on a self-hosted droplet (not serverless), there are no function timeout limits.

### How it works

1. **Upload route** stores the capture with `status: "queued"` and returns 201.
2. **A system cron job** runs every 1-2 minutes and hits `POST /api/captures/process`.
3. **The process route** queries Firestore for captures in a processable state (`"queued"`, `"transcribed"`, `"validated"`, etc.), picks one, and runs the **next** stage for that capture.
4. **Each stage** updates the `status` in Firestore when done. The next cron tick picks up where it left off.

This is a simple state machine driven by cron. Each capture moves through the statuses one stage at a time:

```
queued → transcribing → transcribed → validating → validated → analyzing → profiling → analyzed
                                         ↘ rejected (stop)
```

### Routes

```
POST /api/captures/upload       ← Agent calls this (public, auth required)
POST /api/captures/process      ← Cron calls this (internal, secured by secret)
```

The process route should:
1. Query Firestore for the oldest capture that needs work (status is not `"analyzed"` or `"rejected"`)
2. Determine which stage to run next based on current `status`
3. Run that single stage (since there's no serverless timeout, a stage can take as long as it needs — even a 60-min re-transcription)
4. Update the `status` in Firestore
5. Return — the next cron tick handles the next stage (or the next capture)

If a stage fails, set `status` to `"{stage}_failed"` with an `error` field. The process route can retry failed stages a few times before giving up.

### Cron configuration

Add a system crontab entry on the droplet:

```bash
# Run capture processing every 2 minutes
*/2 * * * * curl -s -X POST http://localhost:3000/api/captures/process -H "Authorization: Bearer $CRON_SECRET" > /dev/null 2>&1
```

Or use `node-cron` inside the Next.js app if you prefer an in-process approach:

```typescript
// lib/cron.ts — import this once at app startup
import cron from "node-cron";
cron.schedule("*/2 * * * *", async () => {
  await fetch("http://localhost:3000/api/captures/process", {
    method: "POST",
    headers: { Authorization: `Bearer ${process.env.CRON_SECRET}` },
  });
});
```

Secure the process route by checking `request.headers.get("Authorization") === "Bearer " + process.env.CRON_SECRET`.

## Processing Pipeline Stages

Each stage updates the `status` field in Firestore so the frontend can show progress.

**Important — merge, don't overwrite:** The platform may already have its own teachable moment categories, analysis logic, and user profiling data from drills and other sources. The capture processing pipeline should **add to and merge with** existing data, never replace it. For example:
- If the user profile already has weaknesses identified from drills, append new capture-sourced weaknesses alongside them (or increase frequency counts on matching patterns) — don't wipe the existing list.
- If the platform already defines teachable moment types, use those categories where they fit. The types listed below (grammar, filler, phrasing, vocabulary, communication) are suggestions — map them to your existing taxonomy if you already have one.
- Drill suggestions from captures should coexist with drill suggestions from other sources (manual curation, drill performance analysis, etc.).

```
Upload (201 returned)
    |
    v
[Stage 1: Re-Transcription]  status: "transcribing"
    |
    v
[Stage 2: Relevance Validation]  status: "validating"
    |                    |
    | (accepted)         | (rejected)
    v                    v
[Stage 3: Deep Analysis]  status: "rejected" (stop)
    |
    v  status: "analyzing"
[Stage 4: User Profiling]
    |
    v  status: "profiling"
[Stage 5: Drill Generation]
    |
    v  status: "analyzed" (done)
```

### Stage 1: Re-Transcription

The agent transcribes locally with Whisper small on CPU. This is accurate enough for filtering (is the user actually talking? is it a real conversation?) but NOT accurate enough for coaching feedback. Misheard words become false "corrections."

**Action:** Re-transcribe the full audio with a paid API.

**Recommended services:**
- **OpenAI Whisper API** — ~$0.006/min. 60-min worst case = $0.36/capture. High accuracy, good with accented English.
- **Deepgram Nova-2** — ~$0.0043/min. 60-min worst case = $0.26/capture. Real-time capable, good diarization.
- **Self-hosted Whisper large-v3** — free after GPU cost. Good option at scale.

**Notes:**
- The audio is stereo (L=mic, R=system). You can split channels before transcription for better speaker attribution, or transcribe the mixed audio and use the existing speaker labels as a guide.
- The agent's local transcript is stored as `agent_transcript` for reference. The server's transcript becomes the authoritative `server_transcript`.
- Store both in Firestore — the agent transcript shows what the local model heard, useful for debugging.

**Update:** Set `status: "transcribing"` before starting, `"transcribed"` when done.

### Stage 2: Relevance Validation

The agent's local 3B LLM already filtered out obvious non-conversations (YouTube videos, music, etc.). But edge cases slip through — the user commenting on a YouTube video, scripted role-play, repetitive English practice that isn't a real conversation.

**Action:** Run a larger LLM (GPT-4o-mini, Claude Haiku, or similar) on the server transcript to validate:
- Is the user genuinely participating in a real conversation with another person?
- Is this an organic, natural conversation (not scripted practice)?
- Is there enough substance for coaching analysis?

**If rejected:** Set `status: "rejected"`, store `rejection_reason`, stop the pipeline. Keep the raw data (don't delete — the user might want to see what was filtered).

**If accepted:** Set `status: "validated"`, continue.

### Stage 3: Deep Analysis

This is the core coaching value. Analyze the server transcript + audio for:

#### Teachable Moments
Specific instances in the conversation where coaching would help:
- **Grammar mistakes** — article omission ("I go to office" vs "I go to the office"), tense confusion, subject-verb agreement, preposition errors
- **Filler words** — excessive "um", "uh", "like", "you know", "basically", "actually" with timestamps and frequency
- **Awkward phrasing** — sentences that are grammatically correct but sound unnatural to a native speaker
- **Missed vocabulary opportunities** — places where a richer word/phrase would be more precise or professional ("good" where "effective" fits better)
- **Communication anti-patterns** — excessive hedging ("I think maybe perhaps we could possibly..."), trailing off mid-sentence, circular explanations

Each teachable moment should reference:
- The transcript line(s) where it occurred (by index or timestamp)
- What the user said
- What would be better and why
- Severity: minor (stylistic), moderate (clarity impact), major (meaning impact)

#### Grammar Patterns
Recurring grammar issues across this and prior captures. Not just individual errors, but patterns:
- "User consistently drops articles before countable nouns"
- "User confuses present perfect and simple past when describing completed actions"
- Track frequency across captures to show improvement over time.

#### Vocabulary Assessment
- Range and sophistication of vocabulary used
- Domain-specific vocabulary (the user might be fluent in technical terms but limited in casual conversation, or vice versa)
- Repeated reliance on simple/vague words ("thing", "stuff", "good", "bad") where more precise alternatives exist

#### Fluency Metrics
- Words per minute (speaking rate)
- Pause frequency and duration (hesitation patterns)
- Self-corrections ("I mean...", "sorry, what I meant was...")
- Response latency (how long after the other person stops speaking does the user start?)

#### Communication Style
- Directness vs hedging
- Confidence indicators (declarative statements vs constant qualifiers)
- Turn-taking behavior (interrupts? long pauses before responding? talks over others?)
- Formality calibration (too formal for the context? too casual?)

**Update:** Set `status: "analyzing"` before starting.

### Stage 4: User Profiling

Build a living profile of the user from each capture. This is NOT just about English ability — it's about understanding the user so the platform can create personalized, relevant drills.

#### Facts to Extract
- **Professional context:** role, company, industry, team structure, what they work on
- **Communication context:** who they talk to (colleagues, clients, external partners), typical formality level
- **Domain knowledge:** what topics they discuss frequently, their expertise areas
- **Personal context:** any facts the user mentions about themselves that could make drills more relatable

#### Speaking Profile (evolves over time)
- **Strengths:** what the user does well (clear technical explanations, good vocabulary in domain X, natural turn-taking)
- **Weaknesses:** recurring issues that appear across multiple captures (not just one-off mistakes)
- **Improvement trajectory:** are specific patterns getting better or worse over time?

Each fact/observation should track its source (capture_id, transcript line) for provenance. When a new capture contradicts an old fact, update it.

**Where to store:** Update the existing `users/{uid}` document (for facts and context) and the existing `skill_memories` collection (for speaking patterns and weaknesses). See "Integration with existing collections" in the data model section. Do NOT create a separate profile document — the platform already has one.

**Merge behavior:** The user profile may already have data from drills and other platform features. This stage must merge capture-sourced insights into the existing data — append new facts, increment weakness frequency counts, add new evidence entries to existing patterns. Never clear existing data to replace it with capture-only data.

**Update:** Set `status: "profiling"` before starting.

### Stage 5: Drill Generation (TBD — needs more design)

The vision is clear but the exact mechanism needs more planning. Two types of drills should eventually be generated from captures:

#### Targeted Drills
Standard platform drills seeded with the user's actual mistakes and context:
- "Practice using articles correctly in technical descriptions" — uses examples from the user's actual speech
- "Reduce filler word usage in explanations" — based on the user's actual filler frequency
- "Expand vocabulary for describing project status" — based on the user's repeated use of simple words in that context

Each drill targets a specific weakness from Stage 3/4 and uses the user's real context (industry, role, conversation topics).

#### Scenario Replay Drills
Recreate the real conversation scenario as a practice drill:
- Use the capture's title, summary, and context to set up the scenario
- The user practices the same conversation but with coaching guidance
- Example: "You were explaining the caching migration plan to your team. Let's practice that explanation with clearer structure and more precise vocabulary."
- The user already knows the context (it was their real meeting), so they can focus purely on improving delivery

These are high-value because the user doesn't need to imagine a scenario — they lived it. The emotional and contextual memory is already there.

**Open design question:** Drills currently live in the `sessions` collection. How exactly captures feed into `sessions` (automatic generation? user-triggered? suggested and user-accepted?) is TBD. For now, store enough analysis data in the capture document that drill generation can be built on top later. The capture's `analysis` field has all the raw material needed.

**Update:** Set `status: "analyzed"` when all stages complete.

## Firestore Data Model

### `captures/{capture_id}` (top-level collection)

Follows the same pattern as `sessions` — top-level collection with a `uid` reference field.

```
{
  // Ownership
  uid: string,                       // Firebase UID — links to users/{uid}

  // Metadata
  status: "queued" | "transcribing" | "transcribed" | "validating" | "validated" | "rejected" | "analyzing" | "profiling" | "analyzed",
  rejection_reason: string | null,
  uploaded_at: timestamp,

  // From agent
  agent_record_id: string,           // The agent's local ID (12-char hex)
  started_at: timestamp,
  ended_at: timestamp,
  title: string,                     // Agent's LLM-generated title
  summary: string,                   // Agent's LLM-generated summary
  agent_transcript: [{               // Agent's local Whisper small transcript
    speaker: string,
    start: number,
    end: number,
    text: string
  }],
  relevant_span: [number, number],   // Agent's hint (treat as rough guide)
  close_reason: string,
  audio_storage_path: string,        // Cloud Storage path

  // Server-generated
  server_transcript: [{              // Re-transcribed with better model
    speaker: string,
    start: number,
    end: number,
    text: string
  }],
  server_title: string,              // Better title from server LLM
  server_summary: string,            // Better summary from server LLM
  duration_secs: number,             // Actual audio duration

  // Analysis (Stage 3)
  analysis: {
    teachable_moments: [{
      type: "grammar" | "filler" | "phrasing" | "vocabulary" | "communication",
      severity: "minor" | "moderate" | "major",
      timestamp: number,             // Seconds into conversation
      transcript_idx: number,        // Index in server_transcript
      user_said: string,
      suggestion: string,
      explanation: string
    }],
    grammar_patterns: [{
      pattern: string,               // e.g. "article omission before countable nouns"
      frequency: number,             // Occurrences in this capture
      examples: [{ transcript_idx: number, text: string }]
    }],
    vocabulary: {
      unique_words: number,
      sophistication_score: number,  // 0-1 scale
      overused_simple_words: [{ word: string, count: number, alternatives: [string] }],
      domain_vocabulary: [string]    // Domain-specific terms used correctly
    },
    filler_words: {
      total_count: number,
      per_minute: number,
      breakdown: { [word: string]: number },  // e.g. {"um": 12, "like": 8}
      timestamps: [number]           // When each filler occurred
    },
    fluency: {
      words_per_minute: number,
      avg_pause_duration_ms: number,
      self_corrections: number,
      avg_response_latency_ms: number
    },
    communication_style: {
      directness: number,            // 0-1 (0 = very indirect/hedging, 1 = very direct)
      formality: number,             // 0-1 (0 = very casual, 1 = very formal)
      confidence: number,            // 0-1 based on qualifier usage
      turn_taking: "balanced" | "passive" | "dominant"
    }
  }
}
```

### Integration with existing collections

The capture processing pipeline should **update existing collections**, not create parallel ones. The platform already has these:

#### `users/{uid}` (already exists)
After analyzing a capture, update the user's existing document with any new facts or context extracted from the conversation (e.g., professional context, domain knowledge, communication patterns). **Merge into existing fields** — do not overwrite what's already there from drills or manual input.

#### `skill_memories/{id}` (already exists)
When capture analysis identifies recurring patterns (grammar issues, filler word habits, vocabulary gaps), feed those into the existing `skill_memories` collection where the coaching memory lives (reinforcements, mastered focus, strengths, weaknesses). This is where the coach "remembers" what to work on — captures are just another signal source alongside drills.

#### `sessions` (already exists) — drill generation TBD
How captures become drills (scenario replay, targeted practice) needs more design work. The vision is clear — a user should be able to practice a scenario from a real recording, and the platform should suggest drills targeting capture-identified weaknesses — but the exact mechanism for creating `sessions` entries from capture analysis is an open design question. For now, store enough data in the capture's `analysis` field that drill generation can be built on top later.

### User-facing features

Captures are not just backend data — users should be able to interact with them in the platform:

- **View captures**: list all their processed captures with title, date, duration, and status
- **Capture detail view**: see the full server transcript (speaker-tagged, timed), analysis results (teachable moments highlighted inline), and summary
- **Audio playback**: play back the original recording (or segments of it) alongside the transcript
- **Scenario replay**: from a capture detail view, the user can launch a drill that recreates the same conversational scenario so they can practice improving their delivery (design TBD — see drill generation note above)
- **Progress tracking**: show how patterns identified in captures change over time (e.g., filler word frequency trending down)
- **Cross-referencing**: future drills can reference specific captures as context ("Based on your meeting from April 10th, let's work on...")

Per-capture processing cost (worst case: 60 minutes of audio):

| Service | Cost/min | 60 min | Notes |

| Deepgram Nova-2 | $0.0043 | $0.26 | Cheaper, real-time capable |

## Security Checklist

- [ ] Verify JWT on every upload request — reject expired/invalid tokens
- [ ] Audio files stored in per-user Cloud Storage paths (`captures/{uid}/...`)
- [ ] Firestore security rules restrict capture access to owning user
- [ ] Audio contains raw conversation data — encrypt at rest (Cloud Storage default)
- [ ] User profile contains PII — apply appropriate Firestore security rules
- [ ] Rate-limit uploads to prevent abuse (e.g., max 20 uploads/hour per user)
- [ ] Set a max upload size (e.g., 50 MB) to prevent abuse
- [ ] Processing pipeline only accesses data for the authenticated user
- [ ] Define audio retention policy (how long raw audio is kept)
- [ ] Async processing has timeout/dead-letter handling for failed stages

## Testing

1. Start your Next.js dev server
2. On the agent side, ensure `.env` has:
   ```
   ELOQUY_AUTH__AUTH_URL=http://localhost:3000/api/auth
   ELOQUY_AUTH__CLIENT_ID=eloquy-desktop
   ELOQUY_AUTH__SERVER_URL=http://localhost:3000
   ```
3. Run `eloquy-agent login` (or the setup wizard)
4. Run `eloquy-agent run` and trigger a conversation (or use `eloquy-agent replay some_audio.wav`)
5. The agent should POST to `/api/captures/upload` after the conversation closes
6. Check Firestore: a document should appear in `users/{uid}/captures/` with `status: "queued"`
7. Check Cloud Storage: `captures/{uid}/{capture_id}/audio.opus` should exist
8. Processing should kick off asynchronously and update the status field through each stage
